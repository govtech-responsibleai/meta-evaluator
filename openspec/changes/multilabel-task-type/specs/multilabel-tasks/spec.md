## ADDED Requirements

### Requirement: Declaring a multi-label task

`EvalTask` SHALL allow a task to be declared multi-label with an exported
`MultiLabelSchema(outcomes: list[str])` wrapper. `task_schemas` SHALL accept
`list[str] | MultiLabelSchema | None`: bare lists retain their existing single-select
meaning, `None` retains its free-form meaning, and only the wrapper denotes multi-label.
A multi-label task's value is a fixed-length ordered vector with one slot per wrapped
outcome — each slot holding either that outcome's name (selected) or the sentinel `"FALSE"`
(not selected). `outcomes` MUST contain at least 2 values; its order defines canonical slot
order. The exact string `"FALSE"` is reserved and MUST NOT appear in wrapped outcomes.

#### Scenario: Valid multi-label declaration
- **WHEN** an `EvalTask` is created with `task_schemas={"response_category": MultiLabelSchema(outcomes=["Clarify","Refuse","Support"])}`
- **THEN** the task is accepted and `response_category` is treated as a 3-slot multi-label vector in that order

#### Scenario: Existing task schema forms remain unchanged
- **WHEN** an `EvalTask` uses a bare outcome list or `None`
- **THEN** the bare list SHALL remain single-select and `None` SHALL remain free-form, with their existing serialized array/`null` shapes unchanged

#### Scenario: Reserved sentinel declared as an outcome
- **WHEN** a `MultiLabelSchema` declares `"FALSE"` as one of its outcomes
- **THEN** construction SHALL raise a validation error explaining that `"FALSE"` is reserved for the not-selected sentinel

#### Scenario: Round-trip serialization preserves slot order
- **WHEN** an `EvalTask` containing `MultiLabelSchema` is serialized and deserialized
- **THEN** the reconstructed schema SHALL remain a `MultiLabelSchema` with the same outcome (slot) order

### Requirement: Judge emits a full ordered vector for multi-label tasks

The judge task class generated by `create_task_class` SHALL type a multi-label task's
field as `list[Literal[*outcomes, "FALSE"]]`, constrained so the list length equals the
number of outcomes and slot `i` is either outcome `i`'s name or `"FALSE"`. A judge using
`structured` or `instructor` answering methods SHALL therefore return the full ordered
vector. Single-select and free-form tasks SHALL be unchanged.

#### Scenario: Structured judge returns a full ordered vector
- **WHEN** a judge with `answering_method="structured"` evaluates a sample for a multi-label task with outcomes `["Clarify","Refuse","Support"]`
- **THEN** the extracted outcome SHALL be a length-3 list where each slot is the corresponding outcome name or `"FALSE"` (e.g. `["FALSE","Refuse","FALSE"]`)

#### Scenario: Wrong-length or out-of-slot value rejected
- **WHEN** a judge response has the wrong number of slots, or slot `i` holds a value that is neither outcome `i`'s name nor `"FALSE"`
- **THEN** the generated model SHALL reject it during parsing

Multi-label tasks SHALL use only `structured` or `instructor` answering methods. Direct
XML configuration SHALL be rejected. When structured-output fallback is enabled, the
fallback sequence for a multi-label task SHALL exclude XML; failure of both supported
methods SHALL surface as an error rather than routing the vector through XML's scalar path.

#### Scenario: Direct XML answering method with multi-label
- **WHEN** an `EvalTask` contains a `MultiLabelSchema` and uses `answering_method="xml"`
- **THEN** construction SHALL raise a clear error stating multi-label requires `structured` or `instructor`

#### Scenario: Multi-label fallback never routes to XML
- **WHEN** a multi-label task enables `structured_outputs_fallback` and its initial JSON-based answering method is unavailable or fails
- **THEN** fallback SHALL try only the other supported JSON-based method and SHALL NOT attempt XML
- **AND** if neither `structured` nor `instructor` succeeds, evaluation SHALL fail clearly

Multi-label outcomes from judge consistency runs SHALL be aggregated independently per
slot and reconstructed in canonical schema order. Each slot SHALL use the existing
consistency convention: the most frequent value wins and a tie resolves to the value that
occurred first across the runs.

#### Scenario: Consistency aggregation votes per slot
- **WHEN** three consistency runs return `["Clarify","FALSE"]`, `["FALSE","Refuse"]`, and `["Clarify","Refuse"]`
- **THEN** the aggregate SHALL be `["Clarify","Refuse"]`

#### Scenario: Consistency tie preserves first occurrence
- **WHEN** two consistency runs disagree on a slot
- **THEN** that slot SHALL use the first run's value, matching existing scalar consistency behavior

### Requirement: Annotators select multiple labels for a multi-label task

The annotation interface SHALL render a multi-label task as a multi-select (checkbox
group) and submit its value as the full ordered vector — each slot holding the outcome
name if checked or `"FALSE"` if unchecked. "Nothing applies" is a valid, fully-defined
submission (all slots `"FALSE"`).

#### Scenario: Selecting multiple labels
- **WHEN** an annotator checks "Clarify" and "Refuse" for a multi-label task with outcomes `["Clarify","Refuse","Support"]` and submits
- **THEN** the stored outcome SHALL be `["Clarify","Refuse","FALSE"]`

#### Scenario: Nothing selected
- **WHEN** an annotator submits a multi-label task with no boxes checked
- **THEN** the stored outcome SHALL be the all-`FALSE` vector of full length (no empty/undefined value)

#### Scenario: Toggling a label off
- **WHEN** an annotator selects then deselects a label
- **THEN** that slot SHALL hold `"FALSE"` in the submitted vector

### Requirement: Multi-label outcomes persist and export faithfully

The annotation API, results storage, and serialization SHALL accept and preserve
vector-valued outcomes (`dict[str, str | list[str]]`). Results metadata SHALL retain the
widened `task_schemas`, including each serialized `MultiLabelSchema`, so standalone load
paths can identify vector columns without an `EvalTask` object. A multi-label task's value
SHALL survive a round trip through JSON, CSV, and parquet exports as the same ordered
vector, with slot order preserved.

#### Scenario: Parquet round-trip
- **WHEN** results containing a multi-label task are saved to parquet and reloaded
- **THEN** the multi-label column SHALL contain the original ordered vector

#### Scenario: CSV round-trip
- **WHEN** results containing a multi-label task are saved to CSV and reloaded
- **THEN** the persisted `MultiLabelSchema` metadata SHALL identify the JSON-encoded column and the values SHALL be recovered as the original ordered vector

### Requirement: Scoring multi-label tasks with classification metrics

A multi-label task scored with `ClassificationScorer` SHALL be binarized — slot `i`
mapped to 1 if it holds outcome `i`'s name and 0 if it holds `"FALSE"`, aligned to schema
order (positional, not merely "non-`FALSE`") — into a single per-item indicator vector of
length equal to the number of outcomes, then scored with the existing `ClassificationScorer`.
The binarize transform SHALL be applied at the scorer call site, gated to
`ClassificationScorer` (where the scorer identity is in scope), AFTER the preprocessing step
has produced the `label` column in canonical schema (slot) order, leaving
`ClassificationScorer` unaware of the `"FALSE"` sentinel. It SHALL NOT live in the
scorer-agnostic preprocessing step, because that would force every scorer onto the indicator
form. Binarize validation (e.g. a foreign slot value) SHALL raise where it is visible, not be
masked into a per-judge NaN by the broad error-handling around scoring.

The scorer's `average` for a multi-label task MUST be `"samples"` or `"macro"`; the existing global default
`"binary"` SHALL be rejected with a clear error naming the two valid choices (it raises in
sklearn on an indicator matrix and would otherwise surface as a silent NaN). Callers SHALL
choose explicitly; `"macro"` is the recommended setting and gives the per-label (per-slot)
average. `"samples"` remains available for per-item overlap when every task in the metric
configuration is native multi-label. The scorer SHALL aggregate across human annotators via
both `individual_average` and `majority_vote`. No dedicated set-overlap scorer is required.

`AltTestScorer` SHALL score the native name vector **unchanged** — it is NOT binarized. Its
`jaccard_similarity` auto-routes on list-valued labels and is designed to consume the name
vector directly; binarizing it would change its scores and is not performed.

#### Scenario: Binarize collapses to one indicator vector
- **WHEN** the vector `["Clarify","FALSE","Support"]` for outcomes `["Clarify","Refuse","Support"]` is binarized
- **THEN** the result SHALL be the single length-3 indicator vector `[1,0,1]` (slot `i` → 1 iff it equals outcome `i`'s name), NOT a one-hot expansion of each slot

#### Scenario: Name-vector requires binarization before sklearn
- **WHEN** an un-binarized name-vector is passed to a metric using `average="samples"`/`"macro"`
- **THEN** it SHALL fail (sklearn target type `multiclass-multioutput` is unsupported), confirming the binarize step is required, not optional

#### Scenario: Per-item overlap with samples averaging
- **WHEN** the expert vector is `["Clarify","FALSE","Support"]` and the judge vector is `["Clarify","FALSE","FALSE"]`, scored with `ClassificationScorer(average="samples")`
- **THEN** the score SHALL reflect the per-item slot overlap between the two binarized vectors (`[1,0,1]` vs `[1,0,0]`)

#### Scenario: Per-label macro averaging
- **WHEN** the same vectors are scored with `ClassificationScorer(average="macro")`
- **THEN** the score SHALL be the unweighted mean of the per-slot (per-label) metric

#### Scenario: All-zero classification score
- **WHEN** an F1, precision, or recall comparison binarizes both expert and judge values to an all-zero indicator vector
- **THEN** the resulting classification score SHALL be `0`

#### Scenario: Default binary averaging is rejected for multi-label
- **WHEN** a multi-label task is scored with a `ClassificationScorer` whose `average` is the default `"binary"` (or any value other than `"samples"`/`"macro"`)
- **THEN** scoring SHALL raise a clear error naming `"samples"` and `"macro"` as the valid choices, rather than failing deep in sklearn and being swallowed into a NaN result

#### Scenario: AltTest scores the native name vector unbinarized
- **WHEN** an `AltTestScorer` scores a multi-label task
- **THEN** it SHALL receive the native name vector (e.g. `["Clarify","FALSE","Support"]`), NOT the binarized indicator vector, and route to its `jaccard_similarity` function unchanged

#### Scenario: Majority vote across annotators
- **WHEN** several humans annotated the same sample and `annotator_aggregation="majority_vote"`
- **THEN** the per-slot majority over the aligned (binarized) vectors SHALL form the consensus vector before comparison, without error

#### Scenario: Even-split slot resolves to label-present
- **WHEN** an even number of annotators is split evenly on a slot under `majority_vote` (e.g. one annotator selects the outcome and one does not, binarized to `1` and `0`)
- **THEN** the per-slot tie SHALL resolve to the selected value (`1`, label-present), so the consensus slot holds the outcome rather than `"FALSE"`. Because binarize runs before the vote, the tie-break operates on integers (`1 > 0`) — a stable "tie ⇒ present" convention — NOT on lexical ordering of the outcome names

### Requirement: Cohen's kappa is unavailable for multi-label tasks

`CohensKappaScorer` SHALL raise a clear error when applied to a multi-label (list-valued)
task, because κ's chance-correction has no statistically valid averaging axis over a
multi-label vector. The error SHALL name `ClassificationScorer` and `AltTestScorer` as the
supported alternatives. Cohen's κ on single-label (scalar) tasks SHALL be unchanged.

#### Scenario: Kappa on a multi-label task errors
- **WHEN** a `CohensKappaScorer` is configured against a task whose schema is `MultiLabelSchema`
- **THEN** scoring SHALL raise a clear error stating κ does not support multi-label tasks and pointing to `ClassificationScorer`/`AltTestScorer`

#### Scenario: Kappa on a single-label task still works
- **WHEN** a `CohensKappaScorer` scores a normal single-outcome task
- **THEN** it SHALL compute κ exactly as before, unaffected by this change

### Requirement: Scoring-strategy selection for native multi-label tasks

A native multi-label task is a single column and SHALL be scored with
`task_strategy="single"` (yielding one multi-label metric value). The existing task-count
validator SHALL continue to reject `task_strategy="multilabel"`/`"multitask"` for a single
task name, preventing a native task from routing into the legacy N-column melt. A native
multi-label task MAY participate in a `multitask` configuration alongside other tasks
(e.g. `[single_class_task, multilabel_task]`), in which case each task is preprocessed
independently through the `SINGLE` path (binarize engaging only for tasks whose schemas
are `MultiLabelSchema`) and the per-task scores are averaged.
When `ClassificationScorer` is used for a mixed scalar + native multi-label `multitask`
configuration, its `average` MUST be `"macro"`; `"samples"` SHALL be rejected because
sklearn does not support it for scalar multiclass input.

#### Scenario: Native multi-label task uses single strategy
- **WHEN** a `MetricConfig` has `task_names=["harm"]` (a multi-label task) and `task_strategy="single"`
- **THEN** scoring SHALL produce one multi-label metric value from the binarized vector

#### Scenario: Wrong strategy for a lone multi-label task is rejected
- **WHEN** a `MetricConfig` has `task_names=["harm"]` and `task_strategy="multilabel"` or `"multitask"`
- **THEN** construction SHALL raise a validation error (a single task name cannot use `multilabel`/`multitask`)

#### Scenario: Multi-label task averaged with a single-class task under multitask
- **WHEN** a `MetricConfig` has `task_names=["toxicity","harm"]` (single-class + multi-label), `task_strategy="multitask"`, and `ClassificationScorer(average="macro")`
- **THEN** each task SHALL be scored independently (binarize applied only to `harm`) and the two scores averaged into one value

#### Scenario: Samples averaging rejected for mixed multitask
- **WHEN** the same mixed configuration uses `ClassificationScorer(average="samples")`
- **THEN** scoring SHALL raise a clear error requiring `average="macro"`
